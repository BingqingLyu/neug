## 二、内容适配策略

核心原则：**同一个素材，不同平台改变的是「切入角度」和「深度」，不是重新写**。

以"Linux 内核 73 万函数调用关系分析"为例：


| 平台       | 切入角度                           | 深度 | 关键元素                   |

## 三、运营方法论

### 1. 内容生产：「一鱼多吃」模式

```
原始素材（完整分析报告）
    ├── 深度长文 → ATA / 公众号 / Medium
    ├── 教程版  → 掘金 / CSDN（加代码）
    ├── 摘要版  → Reddit / Twitter（加截图）
```

**核心动作**：先写一篇最完整的"母文章"，再向下裁剪，而不是分别写。

### 2. 引流闭环设计

每篇文章末尾必须有：

- GitHub 链接（引导 Star）
- 可运行的完整代码（降低尝试门槛）
- 留一个开放性问题（引导评论互动）

### 3. 衡量指标

- **传播效果**：HackerNews 分数、Reddit upvote、知乎点赞
- **转化效果**：GitHub Star 增长、文章发布后的增量
- **质量信号**：评论区的问题深度（越深说明受众越精准）

## 五、开源社区健康度系列文章规划

**数据基础**：report_full.md（完整报告）+ report_short.md（精简版）

**目标平台**：知乎、微信公众号、小红书（每篇覆盖三个平台版本）

### 拆分逻辑说明

两篇报告的视角互补：
- report_full.md：方法论驱动，按维度→排名→赛道→案例展开，数据完整
- report_short.md：结论驱动，先抛反直觉发现，再用对比案例支撑，传播性更强

拆分原则：
- 衰退类案例（LangChain/AutoGen/SD WebUI）合并为一篇，主题聚焦协作网络无法沉淀
- 成功案例（Elasticsearch/PyTorch/OpenClaw）合并为一篇，对比叙事：前两者是经过时间验证的治理范本，OpenClaw是新模式早期样本，结尾留悬念
- 赛道横向对比与技术选型建议合并，目标读者相同——技术选型者

### 最终5篇序列

|:---:|:---|:---|:---|

### 知乎

**受众**：泛技术开发者，有耐心读长文，喜欢有深度的技术分析

**内容形式**：深度长文，基本就是母文章的知乎版本

**切入角度**：技术性问题导向 —— "为什么代码 review 看不出架构问题？"

**关键调整**：

- 标题改成更知乎风格的提问式，如："解剖 OpenClaw：基于代码知识图谱，我们发现了哪些架构债务？"
- 加一个"引发的思考"结尾段：你自己的项目是否有类似的 God Module？
- 其他基本保留。保留 NeuG + CodeScope 操作细节和 Cypher 查询示例（知乎读者愿意 follow 可复现内容）

### 微信公众号

**受众**：泛技术读者，移动端阅读为主，注意力较短，偏向"结论先行"

**内容形式**：图文推送，标题党 + 精简版内容

**切入角度**：结论驱动 —— "一个开源项目 10% 的代码是死代码，而且大家都不知道"

**关键调整**：

- 精简到 3-4 个发现，去掉 Cypher 代码块、调用树数字等技术细节
- 每个发现用小标题 + 一张关键数据表格，结构清晰
- 心跳那个发现（发现2）最适合公众号：有故事性，"加个 if 不就完了？—— 但其实不是"这种反转结构很抓人
- 结尾 CTA：关注/转发，引导到 GitHub

#### 具体调整

**1. 标题**

公众号标题要在微信生态里竞争点击，适合"数字冲击 + 悬念"组合：

* `我们给 OpenClaw 做了一次X 光：bug定位、依赖膨胀、10% 死代码全都藏在这里`
* `OpenClaw代码库体检报告：2000个死函数、34层依赖、一个牵动半个系统的模块`
* `OpenClaw 18000次提交之后：图数据库扫出了哪些肉眼看不见的问题`

**2. 内容精简：保留哪 3-4 个发现**

不是所有发现对公众号读者都同等吸引力，建议保留：

* **发现1**（Bug 扎堆）：有数字、有表格、结论直接
* **发现2**（心跳 bug）：故事性最强，"加个 if 不就完了？——但其实不是"天然适合公众号叙事
* **发现4**（结构热点）：表格直观，"最危险的函数"标签抓眼球
* **发现5**（死代码）：数字大、反常识，适合结尾冲击

发现3（飞书 vs Teams 对比）和发现6（修改密度）技术细节较多，公众号可以去掉。

**3. 删减内容**

* 去掉所有代码块（bash 命令、Cypher 查询）——公众号移动端排版不友好，读者也不期待在这里复现
* 调用树那段（`runHeartbeatOnce` 的树状图）可以**保留但简化**：只保留"穿过7个子系统"的结论，去掉具体函数列表

**4. 结尾 CTA**

公众号结尾标准动作：

* 引导关注/转发（"觉得有用就转给你的技术 leader"）
* GitHub 链接（NeuG + CodeScope）
* 可以加一句互动钩子，但比知乎更轻，不需要长篇开放性问题

### 小红书

**受众**：年轻技术人/程序员，视觉优先，短内容，喜欢"干货总结"卡片

**内容形式**：图文卡片（5-8 张），配简短文字 // todo: 需要设计配图

**切入角度**：强视觉冲击的数字 + 结论 —— "扫了 21000 个函数，发现了这些"

**建议卡片结构**：

1. 封面：惊人数字（"21,057 个函数 / 2,081 个死函数"）
2. 发现1：Bug 扎堆图（模块 bug 次数表格做成可视化）
3. 发现2：心跳 bug 故事（一句话版：想省事加个 flag，结果依赖链已经救不回来了）
4. 发现4：最危险的8个函数（直接搬表格，配注释）
5. 发现5：死代码分布：各模块死函数条形图死代码图（条形图）
6. 总结卡：3个建议，图谱 vs 传统工具（bullet points）
7. 结尾：NeuG/CodeScope 简介 + GitHub 链接

文字风格：

* 每张卡片正文 **不超过3句话**
* 句子短，能断行就断行
* 可以用 emoji 分隔重点（小红书受众接受度高）
* 避免出现代码块、命令行

小红书帖子标题：

* `扫了OpenClaw 2万个函数，发现这些问题👀`
* `OpenClaw代码体检报告｜10%是死代码，还有更惊人的`
* `用图数据库给OpenClaw做X光，结果让我沉默了`（✅）

正文 tag

结尾加话题标签：`#OpenClaw` `#程序员` `#代码质量` `#图数据库` `#开源`#NeuG
